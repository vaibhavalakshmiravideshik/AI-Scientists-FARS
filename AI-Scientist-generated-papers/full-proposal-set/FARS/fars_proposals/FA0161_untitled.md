# Speaker-Attested Grounding for Agent Memory: User-Turn Evidence Filtering Improves False Memory Resistance on HaluMem

## Scope and Constraints

- **Paper Type**: Short paper
- **Target Venues**: NeurIPS, ICML, ICLR, ACL, EMNLP (or similar top AI conferences)

## Introduction

### Context and Motivation

Long-horizon LLM assistants and agents often rely on an external **memory system** to persist information across sessions (e.g., user preferences, events, relationships). These systems typically (i) ingest a dialogue, (ii) write extracted “memory entries” into a store, and (iii) retrieve relevant entries to answer later questions. In practice, a memory error is more damaging than a single-turn generation hallucination because the error can be retrieved and reused repeatedly.

A particularly common failure mode is **assistant-originated false memory**: the assistant says something incorrect, and the memory writer later stores that assistant utterance as if it were a user fact. This is plausible because many memory pipelines treat the entire dialogue transcript as admissible “evidence” when deciding what to store.

**HaluMem** is an operation-level benchmark that directly evaluates hallucinations in memory systems across (a) memory extraction, (b) memory updating, and (c) memory-based question answering (**[HaluMem](./references/HaluMem-Evaluating-Hallucinations-in-Memory-Systems-of-Agents/meta/meta_info.txt)**). HaluMem contains explicit **distractor (interference) memory points** (facts mentioned by the assistant that the user does not confirm), and defines **False Memory Resistance (FMR; higher is better)** as the fraction of those distractors that the system successfully ignores.

### The Problem

Many recent memory systems focus on increasing recall or improving compression, but HaluMem shows that **hallucinated memory writes remain common**. In the HaluMem paper’s Table 3 (HaluMem-Medium), the evaluated systems have moderate-to-low FMR despite often being “conservative” memory writers (e.g., Mem0 FMR 56.80%, Supermemory 51.77%) (**[HaluMem Table 3](./references/HaluMem-Evaluating-Hallucinations-in-Memory-Systems-of-Agents/sections/6.2.1 Overall Evaluation on HaluMem.md)**). The HaluMem GitHub leaderboard further reports that **MemOS** achieves high recall but even lower FMR (44.94% on HaluMem-Medium), suggesting that improved coverage can coincide with increased distractor uptake (**[HaluMem GitHub README](./references/GitHub---MemTensor-HaluMem/sections/README.md.md)**).

A strong closest prior work is **ProMem**, which improves memory extraction recall by generating candidate memories, then “looking back” into the dialogue to verify whether each memory has supporting evidence; if not, the memory is discarded (**[ProMem](./references/Beyond-Static-Summarization-Proactive-Memory-Extraction-for-LLM-Agents/meta/meta_info.txt)**; evidence seeking described in **[ProMem §Evidence seeking](./references/Beyond-Static-Summarization-Proactive-Memory-Extraction-for-LLM-Agents/sections/Evidence seeking and verification..md)**). However, ProMem’s evidence search treats the full dialogue (including assistant turns) as valid evidence. In HaluMem, distractors are constructed to appear in assistant utterances; therefore, a dialogue-wide evidence check can still incorrectly accept an assistant-injected distractor.

### Key Insight and Hypothesis

**Key insight:** For *persistent* memory writes, the dialogue transcript is not a uniformly reliable evidence source. In particular, assistant utterances are generated by a model and can be incorrect; treating them as admissible evidence makes “dialogue-grounded verification” insufficient for rejecting assistant-injected distractors.

**Hypothesis:** If we keep the memory extraction pipeline fixed but restrict the **evidence corpus used to validate memory entries** to **user turns only**, then HaluMem **FMR** (a.k.a. `interference_accuracy` in the official scorer) will increase substantially, because assistant-injected distractors will no longer be “supported”. This will come with some recall loss, but we hypothesize it is small on HaluMem because gold user-relevant memories are primarily stated by the user.

Why this could be wrong: (i) many gold memories might only appear in assistant restatements, so recall collapses; (ii) distractors might be repeated by users (so user-only evidence does not help); (iii) the main effect might be a trivial reduction in total memories rather than a targeted reduction in distractors (we control for this with explicit recall and memory-count reporting).

## Proposed Approach

### Overview

We propose **Speaker-Attested Grounding (SAG)**: when deciding whether to commit a candidate memory entry, require that the memory is supported by **user-turn evidence**, not merely by appearing anywhere in the dialogue.

We intentionally apply this only at the **evidence filtering** stage (not at candidate extraction) to isolate the effect and avoid confounding “what candidates exist” with “what gets accepted”.

### Method Details

We implement a lightweight memory writer that produces a list of extracted memory entries per session (to match HaluMem’s `extracted_memories` interface), then evaluate those extracted entries with HaluMem’s official scoring scripts.

**Inputs (per session).** A HaluMem session contains a list of dialogue turns with `role ∈ {user, assistant}` and `content` (**[HF dialogue schema](https://huggingface.co/datasets/IAAR-Shanghai/HaluMem)**).

**Step 1 — Candidate memory extraction (fixed across all conditions).**
- Prompt an LLM to propose a list of candidate memory entries from the full dialogue transcript.
- Output format: a JSON list of strings, each a single, self-contained memory statement.
- Decoding: `temperature=0`.

**Step 2 — Evidence filtering (the only place conditions differ).**
For each candidate memory string `m`:
1. Construct an **evidence corpus** `E` by concatenating dialogue turns.
   - Baseline (dialogue-wide): `E = all turns (user + assistant)`
   - SAG (ours): `E = user turns only`
2. Ask an LLM to decide if `E` contains sufficient evidence to support `m`.
   - Output format: JSON `{supported: true/false, evidence_quote: "..."}`.
   - If `supported=false`, discard `m`.
   - If `supported=true`, keep `m` (optionally rewritten to be closer to the evidence quote, but **the same prompt and rewrite policy must be used in both conditions**).

This is intentionally a “source-constrained grounding” rule: the only difference is whether assistant text is admissible evidence.

**Headroom checks (deterministic, run before full scoring).**
To avoid spending large API budget on an experiment with little headroom, we run two lexical-proxy checks on a small subset (e.g., 2 users):
1. **Interference headroom:** fraction of `memory_source=="interference"` gold memories whose key content words appear in assistant turns but not in user turns.
2. **Recall floor:** fraction of non-interference gold memories whose key content words appear in user turns (upper bound on achievable recall under user-only evidence).

If interference headroom is low or recall floor is low, we expect SAG to have limited benefit or unacceptable recall loss.

### Key Innovations

1. **Mechanism-level diagnosis:** “Dialogue-grounded verification” can fail when the assistant itself is the source of the distractor; thus, evidence must be source-constrained.
2. **A minimal, testable rule:** restrict evidence to user turns only, changing a single variable while holding the rest of the pipeline fixed.
3. **Benchmark-aligned evaluation:** directly targets HaluMem’s FMR objective (“AI mentions but user does not confirm”) and reports the recall–FMR trade-off.

## Related Work

### Field Overview

**Agent memory systems.** Many LLM agents use external memory to persist user information across sessions, typically implemented as extracted text facts plus retrieval (e.g., MemGPT, MemoryBank, Mem0, Zep, MemOS). These systems differ in representation (plain text vs graphs), write policies (summarize vs extract), and update mechanisms.

**Evaluating memory reliability.** Benchmarks such as LongMemEval and LoCoMo evaluate long-term conversational memory via end-to-end question answering. HaluMem introduces operation-level metrics that separately score extraction, updating, and QA hallucinations, including False Memory Resistance against assistant-injected distractors.

**Write-time quality control.** Several lines of work add write-time checks (self-consistency, verification, contradiction detection, or filtering) to prevent false memories from entering the store. However, most checks treat the dialogue transcript as a single evidence source, without explicitly modeling that assistant utterances can be unreliable.

### Related Papers

- **[HaluMem: Evaluating Hallucinations in Memory Systems of Agents](./references/HaluMem-Evaluating-Hallucinations-in-Memory-Systems-of-Agents/meta/meta_info.txt)**: Introduces operation-level evaluation of memory hallucinations and defines FMR against assistant-injected distractors.
- **[Beyond Static Summarization: Proactive Memory Extraction for LLM Agents (ProMem)](./references/Beyond-Static-Summarization-Proactive-Memory-Extraction-for-LLM-Agents/meta/meta_info.txt)**: Improves extraction by semantic matching and dialogue-wide evidence seeking, but does not restrict evidence to user turns.
- **[MemGPT: Towards LLMs as Operating Systems](https://arxiv.org/abs/2310.08560)**: Uses an OS-like memory manager that swaps information between context and external storage.
- **[MemoryBank: Enhancing Large Language Models with Long-Term Memory](https://arxiv.org/abs/2305.10250)**: Early long-term conversational memory framework with memory writing and retrieval.
- **[LoCoMo: Evaluating Very Long-Term Conversational Memory of LLM Agents](https://arxiv.org/abs/2402.17416)**: Evaluates long-horizon dialogue memory with multi-session reasoning and retrieval.
- **[LongMemEval: Benchmarking Chat Assistants on Long-Term Interactive Memory](https://arxiv.org/abs/2410.10813)**: Evaluates several memory abilities in long-term interactive settings.
- **[Mem0](https://arxiv.org/abs/2409.03340)**: A widely used practical memory system with metadata-enriched memory operations and merging.
- **[Zep: A Temporal Knowledge Graph Architecture for Agent Memory](https://arxiv.org/abs/2501.13956)**: Uses a temporal knowledge graph (Graphiti) for memory organization and retrieval.
- **[MemOS](https://arxiv.org/abs/2507.03724)**: Treats memory as a system-level resource across models and sessions; strong recall but low FMR on HaluMem leaderboard.
- **[ENGRAM: Effective, Lightweight Memory Orchestration for Conversational Agents](https://arxiv.org/abs/2502.01339)**: Proposes lightweight orchestration for memory storage and retrieval in assistants.
- **[A Simple Yet Strong Baseline for Long-Term Conversational Memory of LLM Agents (EMem)](https://arxiv.org/abs/2502.01865)**: Demonstrates strong baselines for memory with simple design choices and careful evaluation.
- **[CogCanvas: Verbatim-Grounded Artifact Extraction for Long LLM Conversations](https://arxiv.org/abs/2601.00821)**: Extracts artifacts with verbatim grounding, related to evidence-backed memory writing.
- **[A-MemGuard](https://arxiv.org/abs/2510.02373)**: Proposes a proactive framework for protecting agent memory from harmful or incorrect content.
- **[Truth-Maintained Memory Agent](https://openreview.net/pdf/b869e47faa435320dcc97159dc4b61c8a917075a.pdf)**: Uses multi-agent write-time quality control to reduce false memory under injected false facts (not evaluated on HaluMem).
- **[Agent Security Bench (ASB)](https://arxiv.org/abs/2410.02644)**: Benchmarks security failures in LLM agents, including long-context vulnerabilities relevant to memory systems.
- **[VIGIL](https://arxiv.org/abs/2601.05755)**: Studies verify-before-commit style safety mechanisms for persistent state (conceptually related to safe memory writes).
- **[DRIFT: Dynamic Rule-Based Defense](https://arxiv.org/abs/2506.12104)**: Explores dynamic rule-based protections relevant to persistent agent state.
- **[The Promptware Kill Chain](https://arxiv.org/abs/2601.09625)**: Analyzes persistent compromise threats, motivating robust memory write policies.
- **[Memory in the Age of AI Agents: A Survey (Forms, Functions, Dynamics)](https://arxiv.org/abs/2512.13564)**: Survey of memory forms and functions in agentic systems.
- **[Graph-based Agent Memory: Taxonomy, Techniques, and Applications](https://arxiv.org/abs/2602.05665)**: Taxonomy of agent memory techniques including graph-based designs.

### Taxonomy

| Family / cluster | Core idea | Representative papers | Benchmarks / evaluation | Known limitations |
|---|---|---|---|---|
| Memory systems (text) | Extract/summarize user info into text memory + retrieval | MemGPT, MemoryBank, EMem, Mem0 | LongMemEval, LoCoMo | Susceptible to false memory writes; unclear provenance |

### Closest Prior Work

1. **HaluMem** (**[paper](./references/HaluMem-Evaluating-Hallucinations-in-Memory-Systems-of-Agents/meta/meta_info.txt)**) defines FMR specifically for “AI mentions but user does not confirm” distractors and provides official scoring code. Our proposal uses HaluMem as the primary evaluation and targets its distractor construction directly.

2. **ProMem** (**[paper](./references/Beyond-Static-Summarization-Proactive-Memory-Extraction-for-LLM-Agents/meta/meta_info.txt)**) introduces dialogue-wide evidence seeking to discard unsupported memories. Our approach is a minimal but meaningful modification: **restrict evidence to user turns**, motivated by the fact that assistant turns can be the source of distractors.

3. **Truth-Maintained Memory Agent** (**[OpenReview PDF](https://openreview.net/pdf/b869e47faa435320dcc97159dc4b61c8a917075a.pdf)**) uses a multi-agent pipeline and credibility signals to reduce false memory under injected false facts. Our work differs in (i) targeting HaluMem’s distractor construction, and (ii) testing a single-variable, speaker-attested evidence rule rather than a large pipeline.

4. **CogCanvas** (**[arXiv:2601.00821](https://arxiv.org/abs/2601.00821)**) grounds extracted artifacts with verbatim quotes. Our approach is complementary: instead of quoting evidence, we restrict which speaker’s text may serve as evidence.

**Novelty Kill Search Summary:** Searched for prior work combining “HaluMem” with role-/speaker-constrained evidence for memory writing (e.g., “HaluMem user-only evidence”, “speaker-attested memory extraction”, “assistant hallucination stored as memory HaluMem”, “role-based memory verification HaluMem”, “user-turn-only memory extraction HaluMem”, and checked HaluMem GitHub issues/README and recent memory-agent surveys). No work was found that explicitly evaluates a **user-turn-only evidence filter** as an intervention for improving HaluMem FMR as of 2026-02-19. (Full query log is in `notes.md`.)

## Experiments

### Experimental Setup

**Benchmark:** HaluMem-Medium (memory extraction task only for the decisive test; update/QA optional).
- Dataset: **IAAR-Shanghai/HaluMem** (**HuggingFace**: https://huggingface.co/datasets/IAAR-Shanghai/HaluMem). License: CC-BY-NC-ND-4.0.

**Evaluation code (official):** HaluMem eval toolkit.
- Eval README: **[eval/README.md](./references/HaluMem-eval-README/sections/Main Content.md)**.
- Scorer code shows FMR is computed from interference memories as the fraction with `memory_integrity_score==0` (named `interference_accuracy`): **[evaluation.py](./references/HaluMem-eval-evaluation-py/sections/Main Content.md)**.

**Conditions (≤3 main conditions):**
1. **Extract-only (prompt baseline)**: candidate extraction from full dialogue; no evidence filtering.
2. **Dialogue-wide evidence filtering (baseline)**: same candidate extraction; evidence corpus includes user+assistant turns.
3. **Speaker-Attested Grounding (ours)**: same candidate extraction; evidence corpus includes **user turns only**.

**Base models (API):**
- Candidate extraction LLM: GPT-4o-mini (deterministic; `temperature=0`).
- Evidence filtering LLM: GPT-4o-mini (deterministic; `temperature=0`).
- Metric judging (official): GPT-4o via the HaluMem evaluation scripts (deterministic; `temperature=0`).

**Subset plan (cost control):**
- Run on **2 users** first (early stop). If FMR gain <0.05 or recall drop >0.10, stop and refute.
- If promising, scale to **5 users** (still far below full 20-user evaluation).

**Resource Estimate**:
- **Compute budget**: negligible GPU if API-only; optionally ≤20 GPU-hours if embedding models are used for optional QA retrieval.
- **API usage (dominant):**
  - HaluMem scoring makes one judge call per (gold memory point) for integrity + per (extracted memory) for accuracy (see `evaluation.py`).
  - To stay within reasonable cost, verification should start with 2-user and 5-user subsets as specified.
- **Infrastructure**: no GUIs/browsers; fully automated.

### Experimental Rigor

- **Determinism / seeds**: All LLM calls are run with `temperature=0` (as recommended by HaluMem eval), so the pipeline should be effectively deterministic; multiple seeds are not required. As a sanity check, re-run one user twice and confirm identical metrics.
- **Main confound (fewer memories → higher FMR)**: Report the **number of extracted memories** and require recall drop ≤0.05 in the success criteria.
- **Headroom check**: Run the dual lexical-proxy headroom checks before spending large judge budget.
- **Early stop**: Stop after 2 users if the effect is clearly too small or recall loss too large.

### Analysis (Optional)

- **Per-source breakdown**: Report FMR on interference memories that appear only in assistant text vs those repeated by the user.
- **Error inspection**: Sample false positives (distractors stored) and false negatives (gold memories dropped) to categorize why user-only evidence fails.

## Success Criteria

**Hypothesis** (directional): Speaker-attested grounding (user-turn-only evidence) improves HaluMem-Medium FMR meaningfully while keeping recall and accuracy close to the dialogue-wide evidence baseline.

**Decision Rule** (concrete — when to stop):
- **Proceed** if, on HaluMem-Medium (5-user subset):
  - FMR improves by **≥ +0.10 absolute** over the dialogue-wide evidence baseline, **and**
  - Recall R decreases by **≤ 0.05 absolute**, **and**
  - The average number of extracted memories does not drop by more than **30%** (to avoid “win by storing nothing”).
- **Pivot** if FMR improves (≥ +0.05) but recall drops >0.05: try a two-tier store (user-attested vs assistant-quoted) as a follow-up idea (not part of this proposal’s main experiment).
- **Refute** if FMR gain is < +0.05 on the 2-user early-stop subset, or if recall drops >0.10.

---

## Impact Statement

If successful, this work provides a simple deployment rule for practitioners building memory layers for chat agents: **do not treat assistant utterances as admissible evidence when committing persistent user memory**. This would reduce the risk of persistent false memories at low engineering cost and provide a benchmark-backed justification for “speaker-attested” memory write policies.
