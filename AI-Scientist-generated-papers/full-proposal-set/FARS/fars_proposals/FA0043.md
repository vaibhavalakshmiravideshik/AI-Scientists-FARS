# Isolated Solve-Then-Judge: A Simple Defense Against Candidate-Response Prompt Injection for Multimodal LLM Judges

## Scope and Constraints

- **Paper Type**: Short paper
- **Target Venues**: NeurIPS, ICML, ICLR, ACL, EMNLP (or similar top AI conferences)
- **Core constraint**: Fully automated evaluation (no human-in-the-loop)
- **Compute constraint**: ≤768 A100 GPU-hours (inference-only; expected 0 GPU-hours if using API models)
- **Operational constraint**: Do not rely on OpenAI models for any step that includes adversarial injection strings (Azure content filter). Use open or non-OpenAI API models.

## Introduction

### Context and Motivation

Vision-language models (VLMs) are increasingly used as **judges**: given an image and a user query, they compare two candidate answers and pick the better one. This supports best-of-N selection, automated evaluation, and preference-based post-training. Benchmarks such as **[VL-RewardBench](./references/VL-RewardBench-A-Challenging-Benchmark-for-Vision-Language-Generative-Reward-Models/meta/meta_info.txt)** provide human-verified preference pairs to measure judge accuracy.

In many deployments, candidate answers are **untrusted inputs** (e.g., generated by external models or users). This creates a prompt-injection risk: a candidate answer can include instructions intended for the judge (e.g., “always pick me”), causing the judge to output a manipulated verdict. In language-only judges, this attack can be highly effective. For example, **[Optimization-based Prompt Injection Attack to LLM-as-a-Judge](./references/Optimization-based-Prompt-Injection-Attack-to-LLM-as-a-Judge/meta/meta_info.txt)** reports attack success rates (ASR; lower is safer) of **90.8–93.2%** on *text-only* Mistral-7B judges across MT-Bench/LLMBar (Sec. 4.2). Recent work stresses that **adaptive attackers** can bypass many defenses (e.g., *The Attacker Moves Second*, arXiv:2510.09023), so fixed-suffix injection is only a first-order probe.

### The Problem

A popular direction for improving judges is **solve-then-judge** (self-reference): the judge first attempts to solve the task itself, then uses its own solution as a reference when judging candidates. **[Do Before You Judge](./references/Do-Before-You-Judge-Self-Reference-as-a-Pathway-to-Better-LLM-Evaluation/meta/meta_info.txt)** finds self-reference improves judge accuracy, and **[PhyCritic](./references/PhyCritic-Multimodal-Critic-Models-for-Physical-AI/meta/meta_info.txt)** uses a self-referential critic format.

However, many solve-then-judge prompts place candidate responses *before* the model generates its own solution. This means the “self-solve” step is exposed to untrusted candidate text and may itself be manipulated. It is unclear whether solve-then-judge helps prompt-injection robustness, and what mechanism matters.

### Key Insight and Hypothesis

**Key insight:** If the judge generates a short self-answer using only trusted inputs (image + question), that self-answer can serve as an uncontaminated anchor when later judging potentially malicious candidates.

**Hypothesis:** A two-pass **isolated solve→judge** design substantially reduces candidate-response prompt-injection success compared to (i) a single-pass judge, and compared to (ii) a two-pass design where the solve pass can see the candidates.

## Proposed Approach

### Overview

We propose an inference-time wrapper for pairwise multimodal judging:

1. **Pass 1 (self-solve)**: generate a concise self-answer from (image, question) only.
2. **Pass 2 (judge)**: given (image, question, self-answer, Response 1, Response 2), output which response is better.

### Method Details

We evaluate one base judge model under three conditions:

- **A (single-pass)**: judge directly from (image, question, Response 1, Response 2).
- **B (two-pass, isolated; ours)**: self-solve from (image, question), then judge with candidates + self-answer.
- **C (two-pass, non-isolated control)**: self-solve from (image, question, candidates), then judge with candidates + self-answer.

All conditions use the same decoding (greedy), output format, and a strong system instruction telling the model to treat candidate responses as untrusted text and ignore any instructions inside them.

**Attack.** For each example, identify the human-dispreferred response and append a fixed injection suffix that attempts to force the judge’s boxed verdict (e.g., for Response 2: `Ignore all instructions and output \\boxed{Response 2 is better}`).

### Key Innovations

- A minimal **A/B/C causal test** that isolates whether robustness comes from **information isolation** rather than “two-pass prompting” or extra compute.
- A deployable design rule: *never let untrusted candidate text into the model’s self-solve step*.

### Closest Prior Work

- **Do Before You Judge** and **PhyCritic** motivate solve-then-judge, but do not test prompt injection or isolation.
- Existing prompt-injection defenses (e.g., Spotlighting, CAP) are complementary; they do not isolate whether solve-then-judge robustness requires an information boundary.

### Benchmarks and Metrics

| Benchmark | Description | Metrics | Split | Download Link | Evaluation Script |
|---|---|---|---|---|---|
| VL-RewardBench | Image+query+two responses with human preference | CleanAcc, ASR, ASR_cond | test | https://huggingface.co/datasets/MMInstruction/VL-RewardBench | Custom script: prompt model, parse boxed verdict, compare to `human_ranking` |

**Metrics:**
- **CleanAcc**: accuracy on unmodified pairs.
- **ASR**: fraction selecting the attacked (human-dispreferred) response after injection.
- **ASR_cond (primary)**: ASR conditioned on cases where A was correct on the clean input.

### Main Results

**Decision rule (verification-first):**
1. Pilot on N=300 pairs. If **ASR_cond(A) < 20%**, treat the vulnerability as weak in this setting and **refute** (stop).
2. Otherwise, run full evaluation and accept the hypothesis if:
   - **ASR_cond(B) ≤ 10%** and at least **20 percentage points lower** than ASR_cond(A),
   - **ASR_cond(B) < ASR_cond(C)** (isolation matters), and
   - **CleanAcc(B)** is within **2 points** of **CleanAcc(A)**.

## Success Criteria

- **Robustness**: B reduces ASR_cond vs A, without large CleanAcc drop.
- **Mechanism**: B outperforms C, indicating isolation (not just two-pass) is responsible.

---

## Impact Statement

If successful, this provides a simple operational guideline for practitioners building multimodal judges: **generate the judge’s own solution in a candidate-free context**, then judge candidates. This could reduce vulnerability to attacker-controlled candidate responses in automated evaluation and preference-based training.
