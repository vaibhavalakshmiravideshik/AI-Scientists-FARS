# Do Farkas Dual Rays Provide Better Repair Hints than IIS for Infeasible LLM-Generated LPs?

## Scope and Constraints

- **Paper Type**: Short paper
- **Target Venues**: NeurIPS, ICML, ICLR, ACL, EMNLP (or similar top AI conferences)
- **Core constraint**: Fully automated evaluation (solver-executed); no human labeling
- **Compute constraint**: Inference-only + solver runs; must fit within the repo budget (≤768 A100-80GB GPU-hours)
- **Solvers constraint**: Use open-source solvers/APIs (avoid commercial Gurobi/CPLEX dependencies)

## Introduction

### Context and Motivation

Large language models (LLMs) are increasingly used to translate natural-language operations research (OR) word problems into executable optimization models. This is attractive because many outputs can be verified automatically by running a solver and checking feasibility and objective value. Recent benchmarks and systems (e.g., MAMO, ORLM, MIND) show that even strong open models still fail frequently on realistic linear programs (LPs) and mixed-integer LPs (MILPs), especially on harder instances.

A common failure mode is **infeasibility**: the generated model has no solution satisfying all constraints. Infeasibility is both common and operationally important: a practitioner cannot even inspect a candidate plan if the model is infeasible, and debugging often requires identifying which constraints are inconsistent.

### The Problem

The most widely used solver-native diagnostic for infeasibility is an **Irreducible Infeasible Subsystem (IIS)**: a minimal set of constraints that is still infeasible. IIS feedback is the primary oracle in recent solver-in-the-loop repair benchmarks and systems (e.g., OR-Debug-Bench, OptiRepair).

However, IIS feedback is **set-valued**: it identifies a subset of constraints but provides no intrinsic ranking. In practice, IIS sets can be too large to include verbatim in an LLM prompt (or constraints are too long), forcing truncation. Once truncated, an unranked IIS list becomes weak guidance: the model must guess which constraint(s) to edit first.

In contrast, linear programming duality provides a classical infeasibility certificate: a **Farkas dual ray** (also called a dual infeasibility certificate). When an LP is infeasible, the solver can return a vector of multipliers over constraints that constitutes a short, checkable proof of infeasibility. Importantly, these multipliers naturally induce a **ranking**: constraints with larger |multiplier| contribute more to the contradiction.

### Key Insight and Hypothesis

**Key insight**: When we cannot show the full IIS (because constraints are long or the IIS is large), the bottleneck is not “knowing the IIS set exists” but **prioritizing which constraints to edit**. A dual-ray certificate provides a principled ranking signal that an unranked IIS does not.

**Hypothesis**: For infeasible LPs generated by an LLM, providing the model with a **Top-K constraint list ranked by Farkas dual-ray magnitude** will improve one-step repair success compared to providing a **Top-K truncation of the IIS set**, under a matched prompt budget.

We could be wrong for two main reasons: (1) IIS sets in this setting may be small enough that ranking is irrelevant; (2) dual rays may be numerically unstable or diffuse, producing a noisy ranking.

## Proposed Approach

### Overview

We propose **DualRayRank**, a training-free feedback interface for solver-in-the-loop optimization-model repair:

1. Generate an optimization model in `.lp` format from a base LLM and solve it.
2. If the model is infeasible, extract a dual infeasibility certificate (dual ray / Farkas multipliers) from an open solver.
3. Convert the certificate into a compact natural-language prompt: a Top-K list of the most implicated constraints (by |multiplier|), optionally with normalized weights.
4. Ask the LLM to output a corrected full `.lp` file in one repair attempt.

### Method Details

#### Task framing
- Input: natural-language word problem.
- Output: full `.lp` file (Objective + Constraints + Bounds + Integrality declarations, if any).
- Verification: solve the `.lp` and check whether the objective matches the ground-truth optimum within tolerance.

#### Solver and certificate extraction
We use **HiGHS**, an open-source LP/MIP solver, via its Python bindings (**highspy**) because it exposes both IIS computation and dual-ray extraction:
- IIS: `getIis()` (with `iis_strategy` enabled)
- Dual ray: `getDualRayExist()` / `getDualRay()`

**Mixed-integer linear program (MILP) note**: MAMO contains both continuous LPs and MILPs. Dual rays are defined for LPs, so when the `.lp` includes integer declarations we compute IIS/dual rays on the **LP relaxation** (drop integrality and re-solve). We only apply DualRayRank when the LP relaxation is infeasible (otherwise there is no dual-ray certificate).

Important solver setting: disable presolve for certificate stability (set `presolve=off`), since presolve can eliminate constraints and make row-level certificates hard to interpret.

#### Prompt construction (deterministic)
Let `K=5`. We create three feedback formats that all include the same (problem text, attempt-0 `.lp`):

- **A. IIS-TopK (baseline)**
  - Compute IIS on the LP relaxation.
  - Sort IIS constraints by **constraint name** (deterministic, independent of solver internal ordering).
  - Provide the first K constraints with their constraint text.

To make “constraint name” well-defined, we instruct the model to name every constraint as `c0001`, `c0002`, ... in the `.lp` output. If constraint names are missing, fall back to file order.

- **B. DualRay-TopK (names only)**
  - Extract dual ray multipliers `y_i` for constraints (rows) on the LP relaxation.
  - Rank constraints by `|y_i|` (descending).
  - Provide the top K constraints with their constraint text.
  - Do **not** include numeric multipliers (tests whether ranking/selection alone helps).

- **C. DualRay-TopK + Weights (ablation)**
  - Same constraint selection as (B).
  - Also include normalized weights `w_i = |y_i| / sum_{j in TopK} |y_j|`.

To control prompt length, each constraint line is truncated deterministically (e.g., first 240 characters after whitespace normalization). All conditions use the same truncation rule.

#### Repair protocol
For each instance:
1. Attempt-0: LLM generates `.lp` file; we solve and record status.
2. If infeasible: run exactly one repair attempt under each condition (A/B/C), producing three repaired `.lp` files; solve each.

### Key Innovations

1. **Dual infeasibility certificates as LLM repair hints**: use solver-produced dual rays (Farkas multipliers) as a structured prompt for optimization-model repair.
2. **Causal separation of ranking vs numeric weights**: compare “dual-ray Top-K names” vs “dual-ray Top-K + weights” to test whether coefficients add value beyond constraint selection.
3. **Open-solver-first design**: use HiGHS to avoid commercial solver dependencies, making the approach easy to reproduce.

## Related Work

### Field Overview

**LLMs for optimization modeling.** Work such as NL4Opt, MAMO, ORLM, OptiMUS, LLMOPT, OptiBench, and MIND evaluates or improves LLMs’ ability to generate optimization formulations and solver-executable artifacts.

**Solver-in-the-loop debugging and repair.** Recent benchmarks (OR-Debug-Bench) and systems (OptiRepair) show large gains from iterative repair using solver-native diagnostics, especially IIS, and from training with solver-verifiable rewards.

**Infeasibility diagnosis tools.** Classical OR practice includes IIS computation, feasibility relaxation, and dual infeasibility certificates (Farkas lemma). Recent LLM systems for infeasibility understanding (e.g., OptiChat) typically use IIS and slack-based explanations, but do not study dual-ray certificates as prompts for automated repair.

## Experiments

### Experimental Setup

**Base model**: Qwen2.5-7B-Instruct (open-weight, widely used in OR modeling work). If unavailable, substitute an equivalently sized open instruct model (e.g., Llama-3.1-8B-Instruct) and report the substitution.

**Solver**: HiGHS via highspy.

**Decoding**: greedy (temperature=0) for determinism.

**Baseline ladder (minimal for this problem)**
- Level 0: attempt-0 only (no repair; greedy decoding)
- Level 0b (inference-time scaling control): attempt-0 best-of-2 (two stochastic samples; e.g., `temperature=0.7, top_p=0.95`; select the first solver-correct one), compute-matched to a 2-call “attempt-0 + one repair” pipeline
- Level 1: IIS-TopK repair (A)
- Level 2: DualRay-TopK repair (B)
- Level 3: DualRay-TopK+Weights (C)

**Resource Estimate**
- **GPU**: ~10–40 A100-80GB GPU-hours (Qwen2.5-7B inference; depends on average prompt+completion length and infeasible rate)
- **CPU**: up to ~4 solver runs per instance (attempt-0 + A/B/C repairs on infeasible cases), plus IIS/dual-ray extraction on infeasible LP relaxations; expect a few CPU-hours with parallelism
- **Engineering**: implement (i) `.lp` parsing + constraint-name→text mapping, (ii) HiGHS solve wrapper with presolve off, (iii) IIS + dual-ray extraction, (iv) deterministic prompt formatting

### Experimental Rigor

- Main A/B/C comparisons use deterministic decoding (no seeds). For the best-of-2 inference-scaling baseline, run ≥3 random seeds and report mean±std.
- Control prompt length with fixed per-constraint truncation and fixed K.
- Sanity check: verify HiGHS returns valid dual rays only when the LP relaxation is infeasible; log cases where infeasible but no dual ray exists (expected rare).
- Data leakage note: MAMO problems may appear in pretraining corpora; since we compare feedback formats on the same instances (no training), leakage primarily affects absolute pass@1, not the relative A/B/C differences.

## Success Criteria

**Hypothesis**: DualRayRank improves one-step repair by prioritizing the most implicated constraints when IIS feedback must be truncated.

**Decision Rule**:
- **Proceed** if, on the subset of infeasible instances with IIS size > K (i.e., truncation regime),
  - (B) improves repair success over (A) by **≥5 percentage points**, and
  - the improvement is also visible in end-to-end Pass@1-after-≤2-attempts (≥2pp).
- **Pivot** if (B) ≈ (A) but (C) > (B) by ≥3pp: interpret as “numbers help, ranking alone doesn’t,” and refine the numeric presentation/normalization.
- **Refute (core claim)** if (B) ≤ (A) (or within ±2pp) on the truncation regime: interpret as “dual rays do not provide a usable ranking signal for current LLM repair.”
- **Refute (practical relevance)** if fewer than **50** infeasible instances fall into the truncation regime (IIS size > K): interpret as “IIS sets are usually small enough that ranking is not a bottleneck,” making dual-ray ranking unnecessary in this benchmark.

---

## Impact Statement

If dual-ray Top-K feedback improves repair, practitioners building solver-in-the-loop optimization modeling systems can use **dual infeasibility certificates** as a compact, principled alternative to unranked IIS lists, improving repair success under prompt budgets without additional training. If it fails, the result is still decision-relevant: it suggests that investments should focus on richer oracles (e.g., IIS with better subset selection, feasibility relaxations) or training-based repair policies rather than dual-ray certificates.
